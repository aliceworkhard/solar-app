# Plan - T034 Time Control Mode Write

## Request

用户希望把协议里的“时控模式”写入 Android BLE 太阳能遥控器 App，当前先做方案供用户确认。

## Current Facts

- 当前 App 使用 BLE 透传，业务协议是 `FF CE + LEN + DESC + CMD + SUB + DATA + 30 + SUM`。
- 当前 5 条 MVP 命令已经固化：`0A` 开/关、`0B` 增加亮度、`0C` 降低亮度、`0D` 读参数、`0E` 读状态。
- 普通业务命令不等待 BLE 回包；notify 到达后被动解析更新 UI。这个策略不能回退。
- `B1` 是参数下载 / 参数回传协议；`MODE=01` 为时控模式。
- 时控模式下载使用 `CMD=B1`、`SUB=00`、`MODE=01`，数据区 21 字节，`LEN=1A`，整帧总长 29 字节。
- 当前 `frameCodec.encodeFrame()` 支持可变长度数据区，理论上可直接编码 `B1` 时控整包。
- 当前 `responseParser.ts` 对 `B1` 模式码的映射与协议速读版冲突：协议写 `01=时控、02=感应/雷达、03=平均`，现代码是 `01=radar、02=time、03=average`。
- 用户补充确认：时控模式不是单字段写入；只要发送一个时控参数写入命令，就必须把整个时控模式的参数一同发送。
- 用户补充确认：为了验证写入结果，用户会在改完时控参数后主动按“读参数”；如果设备回传整个时控模式参数，App 必须把回传值同步到所有对应按钮、步进控件和滑块。
- 用户在 Approval Gate 批注确认：时长 2-5 按累计到达点发送；本次按协议 `LEN=1A` 的 29 字节帧实现，不补齐 40 字节；UI 不采用单独保存按钮，改为每次完成一次参数修改后立即发送整个时控模式参数包；`读参数` 返回 `B1 MODE=01` 时先按同一布局尝试反解并覆盖当前 UI。
- 已读取 `HTML参考/app.html`，其中可参考的交互结构包括：模式按钮、可折叠设置区、时段 tab、滑块、左右步进按钮、最终发送按钮。但 HTML 内容不与当前协议完全一致，不能照搬字段。
- `deviceController.ts` 是集成层，只有本次获批后由当前 agent 单独修改。
- 本地已有未提交的协议 Excel 修改和旧 APK 未跟踪文件；本任务不触碰、不提交这些文件。

## Proposed Scope

本次 T034 做：

- 新增类型化的时控模式参数模型，不让 UI 拼 HEX。
- 新增 `B1 MODE=01` 时控参数编码，输出正式 `FF CE ... 30 SUM` 帧。
- 新增 `DeviceController` 写入入口，例如 `writeTimeControlParams(params)`，通过现有 `dispatchCommand` / `sendWithoutWait` 路径发送。
- 在控制页增加“时控模式”参数编辑区。
- 每个参数控件完成一次修改后，立即收集当前完整时控参数并发送一整包 `B1 MODE=01`；不做单字段写入。
- 控件类型参考 `HTML参考/app.html` 的交互原则：模式/时段用按钮，离散枚举用左右步进按键，连续数值/百分比用滑块或紧凑步进控件。
- 修正 `B1` 参数回传的模式码显示：`01 -> time`，`02 -> radar`，`03 -> average`。
- 新增 `B1 MODE=01` 时控参数回包解析：读参数 notify 到达后，把完整时控参数反解为 `TimeControlParamsDraft`。
- App 接收到时控参数回包后，同步控制页所有对应控件：基础参数、5 个时段的时长/功率、晨亮时间/功率、当前时段摘要、当前 active 时段的滑块位置。
- 增加单元测试覆盖：时控帧长度、字段顺序、累计时长、校验和、校验规则、模式码映射、读参数回包解析、UI 同步、UI 入口不解析 HEX、修改控件后发送整包。

本次不做：

- 不实现感应模式 `MODE=02` 和平均模式 `MODE=03` 参数写入。
- 不把 `B1` 写入改成默认等待回包。
- 不自动在保存后触发 `readParams`，避免引入额外 BLE 行为；用户仍可点“读参数”手动复核。
- 不提供单独“保存时控模式”按钮作为唯一发送入口。
- 不在滑块拖动过程的每一帧连续发送；滑块类控件按一次提交动作发送，例如用户松手、点击轨道或点击步进按钮后发送整包，避免 BLE 被拖动事件刷屏。
- 不照搬 `HTML参考/app.html` 中当前协议没有定义的字段，例如光控开灯电压、光控关灯电压、负载电流等，除非用户后续明确补充这些字段在当前协议中的位置。
- 不修改 Android 原生 BLE、UUID、写入方式、扫描连接策略。
- 不覆盖 `底层协议/新遥控器数据下载与控制协议.xlsx`。
- 不纳入 `项目文件/android-mvp-capacitor/android/.idea/`。
- 不生成正式 APK，除非用户在确认方案时要求打包交付。

## Proposed Protocol Model

时控写入是“整模式参数包”：

- App 内部维护一个 `TimeControlParamsDraft`。
- 用户改任意字段时先更新 draft，再立即按下表把全部字段编码为一整包 `B1 MODE=01` 发送。
- 按钮/步进控件每点击一次算一次修改并发送整包。
- 滑块控件以“提交后的值”为一次修改；拖动过程中只更新显示，松手后发送整包。
- 不提供单字段 `setParam` 入口，也不复用现有未定义的 `setParam(paramId, value)`。

时控数据区按 21 字节编码：

| 数据字节 | 字段 | UI 输入 / 规则 | 编码 |
| ---: | --- | --- | --- |
| 1 | 工作模式 | 固定时控 | `01` |
| 2 | 电池类型 | `1~3` | 原值 |
| 3-4 | 电池电压 | UI 用 V 或预设，内部转 mV | 高低字节 |
| 5-6 | 最大 PWM 占空比 | UI 用百分比 | `round(percent * 255 / 100)` + `00` |
| 7 | 光控延时 | `5~60s`，5 秒步进 | 原值 |
| 8 | 感应灵敏度 | `1~4` | 原值 |
| 9-13 | 时长 1-5 | UI 用每段小时，0.5h 步进 | 30 分钟单位 |
| 14-18 | 功率 1-5 | `0~100%`，5% 步进 | 原值 |
| 19 | 晨亮时间 | 0.5h 步进 | 30 分钟单位 |
| 20 | 晨亮功率 | `0~100%`，5% 步进 | 原值 |
| 21 | 扩展填充符 | 固定 | `00` |

时长编码建议：

- UI 表达为“每段持续时长”，不是直接让用户填 HEX。
- 编码时把第 2-5 段转成累计到达点发送。
- 例如每段分别为 `1h、2h、0.5h、0h、0h`，发送半小时单位为 `2、6、7、7、7`。
- 累计值必须 `0~15`，也就是总时长不能超过 `7.5h`；超出时 UI 阻止发送。

读参数回包同步建议：

- `readParams` 仍沿用现有非阻塞发送策略：点击后只发送 `0D`，不等待 BLE 回包。
- notify 到达后进入 `FrameDecoder`；如果解析到 `CMD=B1` 且 `payload[0] === 01`，按同一 21 字节时控布局反解。
- 反解结果进入 `DeviceController` 的参数状态，例如新增 `timeControlParams` 状态或专门的参数监听器。
- App 层收到参数状态后：
  - 用回包值覆盖本地 `TimeControlParamsDraft`。
  - 如果当前正在控制页，立即刷新所有按键、步进值、滑块进度和 5 段摘要。
  - 如果当前 active 时段不是 1，也保持 active 时段不变，只刷新该时段显示为回包后的真实值。
  - 显示“参数已同步”类结果，但只表示 App 已收到并解析参数，不额外声明设备已应用。
- 如果 `B1` 回包不是 `MODE=01`，本次只更新模式显示，不回填时控控件。
- 如果 `B1 MODE=01` 字节数不足、校验失败或字段越界，记录日志并不覆盖当前 draft，避免把异常 notify 写进 UI。

## Proposed UI Model

参考 `HTML参考/app.html`，但按当前 App 的 2 页 MVP 风格收敛：

- 控制页保留现有“设备状态 / 控制面板”连续布局。
- 控制面板里新增一个紧凑的“时控模式”设置块，默认折叠或半折叠，避免挤压现有开/关和亮度按钮。
- “时控模式”顶层用一个明确按钮进入/选中时控草稿，不把“雷达/时控/平均”只作为只读展示。
- “读参数”仍保留在系统读取预留区；当设备回传 `B1 MODE=01` 后，它会成为同步时控控件的入口。
- 修改任意时控控件后，UI 显示“正在发送整包”/“已发送整包”结果；不再需要额外保存按钮。
- 基础参数：
  - 电池类型：左右步进按键，在 `磷酸铁锂 / 锂电池 / 铅酸` 间切换，并映射到协议 `1~3`。
  - 电池电压：优先按电池类型给预设按钮或步进值，不让用户输入 HEX。
  - 最大 PWM：百分比滑块或 `- / +` 步进，5% 步进。
  - 光控延时：`- / +` 步进或短滑块，5 秒步进，范围 `5~60s`。
  - 感应灵敏度：左右步进按键，`高 / 中 / 低 / 迟顿` 映射到 `1~4`。
- 时段参数：
  - 使用 `时段 1~5` 按钮切换当前编辑段。
  - 每段只显示当前段的“时长”和“功率”，减少一屏信息量。
  - 时长用 0.5h 步进控件；功率用 5% 步进滑块或 `- / +`。
  - 底部提供 5 段摘要，帮助用户确认保存时会发送全部段。
- 晨亮参数：
  - 晨亮时间用 0.5h 步进。
  - 晨亮功率用 5% 步进。
- 发送规则：
  - 每次提交修改前做整包校验；失败时显示哪个字段越界，不发送 BLE。
  - 发送成功只表示 BLE 写入成功；不声称设备已应用。
- 同步反馈：
  - `读参数` 回包解析成功后，控件值以回包为准。
  - 如果用户刚刚修改并发送了一组参数，再手动读参数，回包应覆盖本地 draft，用来判断设备是否真正接受了写入值。
  - 为避免误解，UI 反馈区区分“已发送时控参数”和“已同步设备参数”。

## Risks

- 时长 2-5 已按用户批注采用累计到达点发送。
- 本次已按用户批注采用协议 `LEN=1A` 的 29 字节正式帧，不补 40 字节；如果真机不接受，再基于真实结果调整。
- 电池类型、标称电压、最大 PWM 百分比的产品默认值需要用户确认；默认值只作为 UI 初始值，不应覆盖真实设备参数。
- 时控参数写入通常不会有回包，因此 App 只能确认 BLE 写入成功；是否真正生效需要真机观察或手动读参数复核。
- `responseParser.ts` 模式码修正可能改变读参数后 UI 显示；这是按当前协议速读版纠正，但仍需真机确认。
- 时控参数较多，移动端 UI 容易变长；方案用折叠区、时段按钮和摘要降低密度，但仍需要本地 320/360/390px 检查。
- HTML 参考中有部分字段不在当前协议速读版时控包里；本次不纳入，避免生成协议未定义字节。
- `B1 MODE=01` 回包布局假设与写入布局一致；如果真机回包字段顺序不同，先记录原始 notify，不强行覆盖 UI。
