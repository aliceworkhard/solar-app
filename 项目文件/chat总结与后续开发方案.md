# chat.md 总结与后续开发方案（2026-04-22）

## 1. 结论摘要

- 项目类型本质是“智能硬件本地控制 App”，核心难点在通信层和协议层，不在传统前后端。
- 当前阶段应 **安卓优先**，iOS 暂缓（无 Mac 的现实约束）。
- 已明确通信主线：BLE 连接供应商 IC（黑盒方式适配），2.4G 由硬件侧处理，App 不直接处理射频。
- 目标路径应从“通信打通”开始，再接 UI，而不是先做完整页面。

## 2. chat.md 已确认信息（可直接执行）

### 2.1 现状与约束
- 已有 HTML UI 参考。
- 微信小程序版本已由供应商做完，可作为业务与交互参考。
- Windows 环境可推进安卓开发和调试。
- 供应商 IC 资料有限，需要按黑盒适配思路实施。

### 2.2 BLE 关键参数（已确认 3/4）
- `namePrefix`：`AC632N`（设备名示例 `AC632N_1`）
- `serviceUUID`：`0000FFF0-0000-1000-8000-00805F9B34FB`
- `writeUUID`：`0000FFF1-0000-1000-8000-00805F9B34FB`
- `notifyUUID`：`0000FFF2-0000-1000-8000-00805F9B34FB`
- `writeType`：**未确认**（`WRITE` 或 `WRITE_NO_RESPONSE`）

### 2.3 开发方向（最终共识）
- 先做安卓，不阻塞于 iOS。
- 先做 BLE 调试版 App（扫描、连接、收发、日志），再接业务 UI。
- 协议封装要独立，不允许 UI 直接拼 HEX。
- UUID 与 IC 差异做成 `profile` 配置，不能写死在页面。

## 3. 关键未决项（必须先闭环）

1. `FFF1` 的写入属性确认：`WRITE` / `WRITE_NO_RESPONSE` / 两者都支持。
2. 协议细节确认：帧结构、CRC 算法、ACK/超时/重发、分包粘包规则。
3. 最小命令集固化：至少 5 条（读状态、读版本、开机、关机、一个参数设置）。

## 4. 后续开发总体方案

## 4.1 技术方案
- 推荐：`Capacitor + HTML/CSS/JS + Android 原生 BLE 通信模块`
- 原因：
  - 最大化复用现有 HTML。
  - Windows 可持续推进安卓阶段。
  - 后续补 iOS 时可复用 UI 与协议层设计。

## 4.2 架构分层（强约束）

```text
UI层（页面交互）
  ↓
DeviceController（业务动作）
  ↓
Protocol层（命令构建/回包解析/CRC/缓冲区拆包）
  ↓
BLE层（扫描/连接/订阅notify/写入/重连）
  ↓
设备（AC632N）
```

- UI 只调用业务动作，如 `readStatus()` / `powerOn()`，不直接处理 HEX。
- BLE 层不关心业务字段，只负责稳定传输和状态事件。
- Profile 层管理不同 IC/模组差异：`serviceUUID/writeUUID/notifyUUID/writeType/codec`。

## 4.3 工程目录建议

```text
app/
  ui/
    scan/
    control/
    settings/
    debug/
  ble/
    BleScanner
    BleConnection
    BleNotification
  protocol/
    FrameCodec
    CommandBuilder
    ResponseParser
    CrcUtil
  device/
    DeviceProfile
    DeviceController
    DeviceState
  data/
    LogRepository
    SettingsRepository
```

## 5. 分阶段执行计划

## 阶段 1：通信打通（3-5 天）
- 目标：完成安卓 BLE 调试版。
- 范围：
  - 扫描并连接目标设备。
  - 发现 `FFF0/FFF1/FFF2`。
  - 打开 `FFF2 notify`。
  - 向 `FFF1` 写入并收到回包。
  - 确认 `writeType`。
  - 展示并保存日志。
- 产出：
  - 可运行 APK（调试版）
  - 通信参数确认表（含 writeType）

## 阶段 2：协议层封装（3-7 天）
- 目标：让协议可维护、可复用。
- 范围：
  - 命令构建函数化（至少 5 条核心命令）。
  - 回包解析结构化。
  - CRC 校验实现。
  - 分包/粘包缓冲解析。
  - 超时、重发、错误码处理。
- 产出：
  - 协议模块代码
  - 命令/回包对照文档

## 阶段 3：UI 接入（5-10 天）
- 目标：把真实设备能力接入现有 UI。
- 范围：
  - 复刻主控制页核心交互。
  - 按钮/滑块与命令绑定。
  - 状态回包驱动 UI 刷新。
  - 保留隐藏调试页（原始 HEX、日志导出）。
- 产出：
  - 业务可用安卓版本（内部测试）

## 阶段 4：稳定性与交付（3-5 天）
- 目标：可连续使用、可测试验收。
- 范围：
  - 断线重连。
  - 权限与蓝牙关闭场景处理。
  - 异常恢复（切后台、连接中断、写入失败）。
  - 多机型联调测试。
- 产出：
  - 发布候选包（RC）
  - 测试与问题清单

## 6. 验收标准（执行口径）

### 6.1 通信验收
1. 能扫描并连接目标设备。
2. 能发现 `FFF0/FFF1/FFF2`。
3. 能稳定启用 `FFF2 notify`。
4. 能向 `FFF1` 写入有效命令并收到回包。
5. 连续收发稳定，断线可重连。

### 6.2 协议验收
1. 核心命令发送正确。
2. 回包解析正确，状态映射正确。
3. CRC/校验一致。
4. 分包粘包不导致错帧。
5. 异常包不导致 App 卡死。

### 6.3 工程验收
1. UUID/写入方式不写死在页面，走 profile。
2. 协议层与 UI 层解耦。
3. 日志可查看、可导出。
4. 后续可新增 IC profile。

## 7. 风险与应对

- 风险：`writeType` 未确认导致发送失败。
  - 应对：阶段 1 首先读取 `FFF1` 属性并双模式压测。
- 风险：协议细节不完整（CRC/ACK/分包规则）。
  - 应对：先固化“最小命令集 + 实测回包”作为基线文档。
- 风险：UI 先行导致返工。
  - 应对：坚持“通信与协议优先”，UI 在阶段 3 接入。
- 风险：未来更换供应商 IC。
  - 应对：从第一版起启用 profile + codec 机制。
